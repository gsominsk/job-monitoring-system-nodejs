# Уточняющие вопросы для финализации спеки

## Критические решения

### 1. Статистические паттерны (REQ-005)

**Контекст**: Нужно проанализировать минимум 3 характеристики jobs. В спеке я предложил 4 примера, но нужно выбрать финальный набор.

**Вопрос**: Какие конкретно паттерны анализировать?

**Предложенные варианты**:

**Категория: Naming patterns**
- [ ] Job name содержит uppercase буквы (А-Я, A-Z)
- [ ] Job name содержит цифры
- [ ] Job name начинается с определенного префикса (например "critical-", "batch-", "test-")
- [ ] Длина имени (buckets: короткие <10, средние 10-20, длинные >20)
- [ ] Job name содержит дефисы/underscores
- [ ] Job name в camelCase vs kebab-case vs snake_case

**Категория: Arguments patterns**
- [ ] Количество аргументов (0, 1-2, 3+)
- [ ] Наличие специфичных флагов ("--fast", "--quality", "--debug")
- [ ] Общая длина всех аргументов (в символах)
- [ ] Аргументы содержат пути к файлам (содержат `.` или `/`)
- [ ] Первый аргумент vs последующие (разная семантика?)

**Категория: Temporal patterns**
- [ ] Время суток (buckets по часам: ночь, утро, день, вечер)
- [ ] Jobs в burst'ах (>5 jobs за 10 секунд)
- [ ] Корреляция между временем выполнения и успехом
- [ ] День недели (будни vs выходные) — если приложение долго работает

**Категория: Execution patterns**
- [ ] Retry rate correlation (jobs которые требовали retry vs не требовали)
- [ ] PID четный vs нечетный (абсурдный паттерн, но может быть забавным)
- [ ] Порядковый номер job (первые 10 vs остальные — "warmup effect")

**Рекомендация**: Выбрать 4-5 паттернов из разных категорий для разнообразия.

### 2. Retry стратегия (Open Question #1)

**Вопрос**: Retry делать сразу или с задержкой?

- **Вариант A (immediate)**: Job упал → сразу retry → быстрая обратная связь
  - ✅ Проще реализация
  - ✅ Быстрее видим результат
  - ❌ Нереалистично для production (transient failures нужно переждать)

- **Вариант B (delayed, 500ms-1s)**: Job упал → wait 500ms → retry
  - ✅ Более реалистично (time for resource recovery)
  - ✅ Избегает "thundering herd" если много jobs упали одновременно
  - ❌ Чуть сложнее (нужен setTimeout)

**Моя рекомендация**: Вариант A для MVP (проще), но сделать конфигурируемым через константу `RETRY_DELAY_MS = 0` (легко изменить на 500).

### 3. Ограничения на хранение jobs (Open Question #2)

**Контекст**: In-memory storage → память растет бесконечно.

**Вопрос**: Нужен ли лимит на количество хранимых jobs?

- **Вариант A (no limit)**: Храним все jobs до перезапуска
  - ✅ Проще
  - ✅ Для тестового задания адекватно
  - ❌ Потенциальная утечка памяти (если долго работает)

- **Вариант B (FIFO limit, e.g. 1000 jobs)**: Oldest jobs удаляются
  - ✅ Защита от утечки памяти
  - ❌ Теряем старую статистику
  - ❌ Усложняет код

- **Вариант C (time-based, e.g. purge >1 hour old)**: Автоочистка старых
  - ✅ Сбалансированно
  - ❌ Нужен periodic cleanup job

**Моя рекомендация**: Вариант A для MVP, задокументировать как known limitation в README.

---

### 4. Лимит параллельных процессов (Open Question #3)

**Контекст**: Операционная система имеет лимиты на количество процессов (Windows: handle limit, Linux: ulimit).

**Вопрос**: Нужно ли искусственно ограничивать concurrency?

- **Вариант A (no cap)**: Доверяем OS
  - ✅ Проще
  - ✅ Максимальная производительность
  - ❌ Может привести к OS-level ошибкам (hard to debug)

- **Вариант B (soft cap, e.g. 100)**: Queue дополнительные jobs
  - ✅ Предсказуемое поведение
  - ✅ Защита системы
  - ❌ Нужна очередь + queueing logic

**Моя рекомендация**: Вариант A для MVP, добавить в README рекомендацию "не запускать >100 одновременно без тестирования".

## Дополнительные уточнения

### 5. Логирование stdout/stderr процессов

**Вопрос**: Нужно ли захватывать вывод dummy-процессов?

**Контекст**: Dummy scripts могут писать в консоль (echo/printf).

- **Вариант A (ignore)**: Не сохраняем stdout/stderr
  - ✅ Проще
  - ✅ Для dummy-процессов не критично
  
- **Вариант B (capture)**: Сохраняем в job metadata
  - ✅ Полезно для debugging реальных процессов
  - ❌ Больше памяти
  - ❌ В спеке это marked as "out of scope"

**Моя рекомендация**: Вариант A (ignore), это явно в "Out of Scope".

### 6. Формат конфигурации

**Вопрос**: Как передавать настройки в приложение?

**Опции**:
- Environment variables (PORT, LOG_LEVEL, etc.)
- JSON config file (config.json)
- Command-line arguments (--port=3000)
- Hardcoded defaults with override via env

**Моя рекомендация**: Hardcoded defaults + environment variables для production overrides.

Пример:
```javascript
const PORT = process.env.PORT || 3000;
const MAX_RETRIES = parseInt(process.env.MAX_RETRIES, 10) || 1;
const LOG_LEVEL = process.env.LOG_LEVEL || 'info';
```

### 7. Тестовые данные для демо

**Вопрос**: Нужен ли seed script для генерации тестовых jobs?

**Контекст**: Для демонстрации статистики хорошо бы иметь быстрый способ создать 50-100 jobs.

**Вариант A**: Ручное тестирование через curl/Postman  
**Вариант B**: Seed script (Node.js), который делает POST requests

**Моя рекомендация**: Добавить простой `scripts/seed.js`:
```javascript
// Send 100 random jobs to localhost:3000
for (let i = 0; i < 100; i++) {
  fetch('http://localhost:3000/jobs', {
    method: 'POST',
    body: JSON.stringify({
      jobName: generateRandomName(),
      arguments: generateRandomArgs()
    })
  });
}
```

Это не обязательно, но сильно упростит тестирование.
